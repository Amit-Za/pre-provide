# מדריך הצגת ProvIDe - פלטפורמת אימות זהות בנקאית
## 20-30 דקות | הצגה מקצועית וממוקדת

---

## מבנה ההצגה (20-30 דקות)

### 1. פתיחה והצגת הבעיה (3-4 דקות)

**נקודות מפתח:**
- הצגת הבעיה: תהליכי KYC מסורתיים יוצרים חיכוך גבוה (30-40% נטישה)
- אבטחה: אחסון מסמכי זיהוי יוצר סיכוני אבטחה ופריצות
- כפילות: משתמשים מאומתים שוב ושוב בכל שירות
- פער בשוק: בנקים כבר ביצעו זיהוי מחמיר, אך לא משמשים כספקי זהות

**הודעות מרכזיות:**
- "כיום, כל שירות דורש העלאת מסמכים מחדש - גם אם המשתמש כבר מאומת ברמה בנקאית"
- "הבנק הוא הגורם המאומת ביותר, אך לא משמש כספק זהות דיגיטלית"

---

### 2. הפתרון - ProvIDe (4-5 דקות)

**הצגת החזון:**
- פלטפורמה מרכזית המתווכת בין בנקים לשירותים דיגיטליים
- כפתור הזדהות אחיד: "אימות זהות באמצעות הבנק"
- הבנק כעוגן אמון (Trust Anchor) - לא מחליף KYC בנקאי, אלא משתמש בו

**תהליך העבודה (Flow):**
1. משתמש לוחץ על כפתור "אימות זהות באמצעות הבנק" בשירות הדיגיטלי
2. מעבר מאובטח לבנק דרך OpenFinance.ai
3. אימות חזק מול הבנק (סיסמה, OTP, ביומטריה)
4. הבנק מאשר זהות ומחזיר אישור דיגיטלי (Identity Assertion)
5. השירות מקבל אישור מאומת - ללא מסמכים

**הודעות מרכזיות:**
- "אפס חיכוך - אין צורך בהעלאת מסמכים"
- "אבטחה בנקאית - משתמשים בתשתית קיימת ומאומתת"
- "תואם רגולציה - מבוסס על סטנדרטים תעשייתיים"

---

### 3. ארכיטקטורה ואבטחה (6-8 דקות)

#### 3.1 ארכיטקטורה כללית

**שלושה רכיבים מרכזיים:**
1. **צד לקוח (Client App)**: Angular/React, ממשק RTL, ניהול Redirects
2. **פלטפורמת ProvIDe**: מיקרו-שירותים מבוססי Node.js, Express.js, PostgreSQL, Prisma
3. **שכבת אימות בנקאי**: OAuth 2.0 דרך OpenFinance.ai

**תרשים זרימה:**
- משתמש → אפליקציית לקוח → ProvIDe → בנק → ProvIDe → אפליקציית לקוח → שירות דיגיטלי

**טכנולוגיות:**
- **Backend:** Node.js 18+, Express.js
- **Database:** PostgreSQL 15+ עם Prisma ORM
- **Caching:** Redis (token blacklist, rate limiting, session storage)
- **Message Queue:** RabbitMQ (תקשורת אסינכרונית בין מיקרו-שירותים)
- **Infrastructure:** Docker, NGINX (load balancing), Loki (logging), Monitoring (TBD)

#### 3.2 אבטחה - עקרונות מרכזיים

**1. OAuth 2.0 / OpenID Connect:**
- סטנדרט תעשייתי מאומת
- Authorization Code Flow עם PKCE
- טוקנים קצרי חיים עם Refresh Tokens
- חתימה קריפטוגרפית (JWT/JWS)

**2. הפרדת נתונים ומבנה Database:**
- **המערכת לא שומרת:**
  - מסמכי זיהוי
  - פרטי התחברות לבנק
  - סיסמאות או נתונים פיננסיים
  - PII (Personal Identifiable Information)
- **המערכת שומרת רק:**
  - מטא-נתונים של אימות (סטטוס, תאריך, bank_user_id מוצפן)
  - הרשאות והסכמות משתמש (authorizations table)
  - טוקנים (רק hash, לא הטוקן עצמו)
  - Audit logs לצורכי רגולציה

**מבנה Database (PostgreSQL + Prisma):**
- **users:** רק סטטוס אימות (verification_status), bank_user_id מוצפן, לא PII
- **services:** רישום שירותים דיגיטליים (client_id, client_secret מוצפן, redirect_uris)
- **authorizations:** הסכמות משתמש (consent records) - scopes, consent_given_at, consent_revoked_at
- **tokens:** ניהול טוקנים (access_token, refresh_token, id_token) - רק hash, לא הטוקן עצמו
- **audit_logs:** לוגים לצורכי compliance - immutable, GDPR-compliant retention
- **banks:** הגדרות בנקים ואינטגרציה (OAuth endpoints, OpenFinance.ai IDs)

**עקרונות עיצוב:**
- לא שומרים PII - רק מטא-נתונים
- טוקנים: רק hash (SHA-256) למעקב revocation
- הסכמות: מעקב מלא של consent lifecycle
- Audit: לוגים immutable לצורכי compliance

**3. הצפנה ואבטחה:**
- **TLS 1.3:** כל התקשורת החיצונית
- **AES-256-GCM:** הצפנת נתונים במנוחה (Encryption at Rest)
- **Bcrypt (cost 12):** הצפנת client secrets
- **SHA-256:** Hash של טוקנים למעקב revocation
- **RSA-2048 / ECDSA-P256:** מפתחות חתימה ל-JWT
- **JWT/JWS:** חתימה דיגיטלית על Identity Assertions
- **JWKS:** ניהול מפתחות מאובטח, rotation אוטומטי כל 90 יום
- **Node crypto:** ספריית הצפנה מובנית

**4. מיקרו-שירותים (Microservices):**
- הפרדת אחריות (Separation of Concerns)
- **OAuth Authorization Server:** ניהול OAuth 2.0 / OpenID Connect flows, הנפקת טוקנים
- **Identity Assertion Engine:** יצירת JWT assertions עם חתימה קריפטוגרפית
- **Consent Orchestration Service:** ניהול הסכמות משתמש, scope management
- **OpenFinance.ai Integration Layer:** חיבור מאובטח לבנקים דרך OpenFinance.ai
- **Client Registry Service:** רישום ואישור שירותים דיגיטליים
- **Audit & Compliance Module:** לוגים, audit trails, תאימות רגולטורית

**תקשורת בין שירותים:**
- **RabbitMQ:** תקשורת אסינכרונית (audit logs, token revocation, bank callbacks, notifications)
  - **Queues:** audit.events, token.revocation, bank.callbacks, notifications, consent.events
  - **Patterns:** Pub/Sub (broadcasts), Work Queues (processing), Routing, Dead Letter Queues
  - **Benefits:** Decoupling, reliable delivery, auto-retry, scalability
- **HTTP/REST:** תקשורת סינכרונית (OAuth flows, service registration, consent screens)
  - Express.js REST APIs
  - OAuth 2.0 endpoints
  - Service management APIs

#### 3.3 פרטיות (Privacy by Design)

**1. עקרון מינימום נתונים (Data Minimization):**
- שירות מקבל רק את הנתונים שהוא צריך
- לא מעבירים מסמכים - רק אישורים מאומתים
- גישה מבוססת הרשאות (Scope-based)

**2. הסכמה מדעת (Informed Consent):**
- מסך הסכמה ברור לפני כל שיתוף
- המשתמש רואה בדיוק מה יועבר
- אפשרות לבטל הסכמה בכל עת

**3. שקיפות:**
- המשתמש יכול לראות אילו שירותים מחוברים
- Dashboard לניהול הרשאות
- Audit trail מלא

**4. תאימות רגולטורית:**
- GDPR compliant
- עמידה בדרישות רגולציה פיננסית
- Audit logs לצורכי בדיקה

**הודעות מרכזיות:**
- "אנחנו לא שומרים מסמכים - רק מאשרים זהות"
- "הבנק לא חושף נתונים פיננסיים - רק מאשר זהות"
- "המשתמש שולט - רואה ומאשר כל שיתוף"

### 4. Use Cases (3-4 דקות)

**1. שירותי הלוואות ופיננסים:**
- החלפה מלאה של העלאת מסמכים
- פתיחת חשבון תוך שניות
- הפחתת נטישה ב-30-40%

**2. פלטפורמות מסחר:**
- אימות זהות לקוחות ומוכרים
- הפחתת הונאות זהות
- פתיחת חשבונות מאומתים

**3. שירותים רגישים:**
- ביטוח, קריפטו
- שירותים מבוססי גיל/אזרחות
- שירותים עתירי רגולציה

**דגש:** כל שירות יכול להטמיע כפתור אחד - אינטגרציה פשוטה

### 6. סיכום ושאלות (2-3 דקות)

**נקודות סיכום:**
- פתרון מודרני המבוסס על תשתית קיימת
- אבטחה בנקאית + פרטיות מקסימלית
- הפחתת חיכוך + תואם רגולציה
- פוטנציאל להפוך לסטנדרט תעשייתי

**הכנה לשאלות נפוצות:**

**Q: מה אם הבנק לא תומך?**
A: אנו עובדים דרך OpenFinance.ai המחברת לבנקים מרכזיים. ככל שיותר בנקים יתחברו, הפתרון יהפוך נגיש יותר.

**Q: מה עם פרטיות?**
A: המערכת לא שומרת מסמכים או נתונים פיננסיים. רק מטא-נתונים של אימות. כל שיתוף דורש הסכמה מפורשת.

**Q: איך זה שונה מ-Open Banking?**
A: Open Banking מתמקד בנתונים פיננסיים. אנחנו מתמקדים באימות זהות - הבנק מאשר זהות, לא חושף נתונים.

**Q: מה עם רגולציה?**
A: מבוסס על OAuth 2.0/OpenID Connect - סטנדרטים מאומתים. עומד ב-GDPR ודרישות רגולטוריות פיננסיות.

**Q: איך השירותים מתחברים?**
A: אינטגרציה פשוטה - כפתור אחד, OAuth flow סטנדרטי. API RESTful עם תיעוד מלא. השירותים מקבלים client_id ו-client_secret לאחר רישום ואישור.

**Q: מה הטכנולוגיות שבשימוש?**
A: Backend: Node.js + Express.js, Database: PostgreSQL + Prisma ORM, Caching: Redis, Message Queue: RabbitMQ. Infrastructure: Docker, NGINX, Loki (logging), Monitoring (TBD).

**Q: איך מיקרו-שירותים מתקשרים?**
A: שילוב של HTTP/REST לתקשורת סינכרונית (OAuth flows) ו-RabbitMQ לתקשורת אסינכרונית (audit logs, token revocation, notifications). זה מאפשר decoupling ו-scalability.
